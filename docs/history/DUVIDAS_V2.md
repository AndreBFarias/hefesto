## 1. PENDÊNCIAS DIRETAS (precisam decisão)

### 1.1 Seção 1 da auditoria original — fechar o loop

O próprio `HEFESTO_DECISIONS.md` admite (linha 7): "seção 1 (Contradições) da auditoria não foi incluída no review. Pendente."

**1.2** — `check_anonymity.sh` cobre só `claude|anthropic|openai|chatgpt|gpt-[0-9]|by AI|AI-generated`. A REGRA -1 proíbe também `Opus`, `Sonnet`, `Gemini`, `Copilot`, `by Claude`, `Generated by`, `Feito por`, `Criado por`, `Autor:`.
  - **Proposta:** regex expandido:
    ```bash
    FORBIDDEN='claude|anthropic|openai|chatgpt|gpt-[0-9]|opus|sonnet|gemini|copilot|by (claude|ai)|(ai|machine)-generated|generated by|feito por|criado por|autor:'
    ```

**1.3** — Filtro `ALLOWED_CONTEXT='api_key|provider|model|config|client|engine'` gera falso negativo crítico: linha com "Claude config here" passa porque tem "config".
  - **Proposta:** remover o whitelist. Exceções viram lista explícita de arquivos ignorados (ex: `LICENSE`, `NOTICE` quando atribuir a libs upstream). Alternativa: regex contextual tipo `\b(api_key|provider|model)\s*=` só casa quando a palavra é parte de expressão técnica, não prosa.

### 1.2 Renomear `CLAUDE.md` da árvore de arquitetura

O `RESPOSTAS.md` (linha 5) diz "vamos excluir esse arquivo depois de criada as sprints iniciais". O `HEFESTO_DECISIONS.md` pergunta (linha 347): "Confirma renomear `CLAUDE.md` pra `AGENTS.md` no repo?"

- **Dúvida:** excluir (risco de perder contexto pra IA futura) ou renomear para `AGENTS.md` (convenção recente, mantém contexto)? Minha recomendação: `AGENTS.md`, porque o arquivo é útil e só o nome viola a regra.

### 1.3 Nome do service na TUI offline (decisão 2.8)

TUI mostra botão `[Iniciar daemon]` que chama `systemctl --user start hefesto`. Mas a decisão 3.4 cria DUAS units: `hefesto.service` (graphical) e `hefesto-headless.service`. A TUI não tem como adivinhar qual está instalada.

- **Proposta:** TUI detecta via `systemctl --user list-unit-files 'hefesto*.service'` e escolhe a única habilitada. Se ambas ou nenhuma, mostra dropdown.

### 1.4 Hotkey via botões do controle vs gamepad virtual (decisão 3.3)

A decisão 3.3 diz "hotkey via botões do controle é default (passa pelo daemon, não precisa de input group)". Mas o Sprint 6.3 cria gamepad virtual via uinput. Nesse modo, o controle físico envia input que o daemon lê E o uinput replica pro jogo. Se hotkey é combo de botões do controle, ela também vai pro jogo.

- **Dúvida:** hotkey em modo emulação exige que o daemon intercepte (bloqueie) os botões do combo? Isso introduz input lag e conflita com jogos que usam o mesmo combo. Ou a hotkey só funciona quando a emulação está desligada?
- **Proposta:** hotkey em modo emulação usa combo "sagrado" (ex: PS button + D-pad) que nunca é repassado ao uinput. Documentar a reserva no README.

### 1.5 uinput: setfacl vs udev rule

A decisão 3.2 diz "Regra ACL pra `/dev/uinput` grupo do usuário ou acesso via setfacl". São soluções diferentes:

- **setfacl** não persiste reboot a menos que esteja em script de login.
- **udev rule** é persistente, mas `/dev/uinput` é char device criado pelo módulo, não por USB enumeration — precisa `KERNEL=="uinput"` com `TAG+="uaccess"`. Funciona.

- **Proposta:** udev rule em `/etc/udev/rules.d/71-uinput.rules`:
  ```
  KERNEL=="uinput", SUBSYSTEM=="misc", MODE="0660", TAG+="uaccess"
  ```
  Mesma lógica do `uaccess` já adotado pros hidraw. Remover menção a setfacl.

### 1.6 PT-BR em logs técnicos

HEFESTO_DECISIONS.md pergunta (linha 348). Três posições plausíveis:

- **PT-BR total** — consistente com CLAUDE.md global, logs espelham UI.
- **EN total** — logs técnicos são pra desenvolvedor/debug; `grep "connection failed"` em StackOverflow é universal.
- **Híbrido** — nível `INFO`+ em PT-BR (pro usuário ver no `hefesto daemon logs`); `DEBUG` em EN (pra dev).

- **Recomendação:** híbrido. Mensagens que o usuário final vê em `systemctl status` / `journalctl -u` em PT-BR; stack traces e `DEBUG` em EN porque são pra desenvolvedor.

## 2. INCONSISTÊNCIAS NOVAS INTRODUZIDAS PELOS PATCHES

### 2.1 `ControllerState.transport` ausente no Patch 2

Patch 2 (linhas 114-120) define:

```python
@dataclass(frozen=True)
class ControllerState:
    battery_pct: int
    l2_raw: int
    r2_raw: int
    connected: bool
```

Patch 7 (ADR-008, linha 298) diz: "`ControllerState` marca `transport: Literal['usb', 'bt']`".

- **Correção:** adicionar `transport: Literal["usb", "bt"]` no Patch 2 antes de executar W1.1.

### 2.2 `MatchCriteria.matches()` retorna `False` para fallback

Patch 3 (linha 197): `return all(conditions) if conditions else False`.

Se `fallback.json` tem `match: {}` (tudo vazio) pra casar qualquer janela, esse código retorna **False** — o fallback nunca casa.

- **Correção A:** `return all(conditions) if conditions else True` (vazio = wildcard).
- **Correção B:** classe separada `MatchAny` com `matches() → True`, usada só em `fallback.json`. Schema fica mais explícito.
- **Recomendação:** B. Evita que um perfil com match vazio por engano vire wildcard silencioso.

### 2.3 `exe_name`: `/proc/PID/comm` vs `/proc/PID/exe`

Patch 3 referencia `window_info["exe_name"]` casando com `"Cyberpunk2077.exe"` (17 chars).

- `/proc/PID/comm` trunca em 15 chars (`Cyberpunk2077.e`). Não casa literal do spec.
- `/proc/PID/exe` é symlink pra `/home/.../Cyberpunk2077.exe` — precisa `os.path.basename(os.readlink(...))`.

- **Correção:** Sprint 6.1 usa `/proc/PID/exe` via `readlink`. Patch 3 e spec original linha 351 precisam refletir que `process_name` casa com basename do exe real, não `comm`.

### 2.4 `window_title_regex`: `re.search` vs `re.fullmatch`

Patch 3 linha 194: `re.search(self.window_title_regex, window_info["wm_name"])`.

Spec original (linha 351) tem exemplo `.*Cyberpunk.*`. Com `re.search`, os `.*` são redundantes (qualquer substring casa). Com `re.fullmatch`, são necessários.

- **Dúvida:** padronizar em `search` (mais conveniente, menos surpresa pro usuário escrever perfil) e remover `.*` do exemplo? Ou `fullmatch` (mais estrito, exige pensar)?
- **Recomendação:** `search`. Menos fricção pra criador de perfil. Atualizar o exemplo canônico no spec.

### 2.5 `RateLimiter.per_ip` sem eviction

Patch 4 (linha 215): `per_ip: dict[str, deque[float]] = defaultdict(lambda: deque(maxlen=RATE_PER_IP))`.

O dict cresce indefinidamente: cada IP novo adiciona entrada e nada remove. Ataque trivial ou só muitos mods = vazamento de memória.

- **Correção:** após o cleanup do `ip_window`, se `len(ip_window) == 0`, `del self.per_ip[ip]`. Memória volta ao baseline quando IPs somem.

### 2.6 `hefesto-headless.service` `WantedBy=default.target`

Patch 8 (linha 325): `WantedBy=default.target`.

Em systemd `--user`, `default.target` é symlink que pode apontar pra `graphical-session.target`, `multi-user.target` ou ser customizado. Em sessão gráfica, isso pode disparar o service antes de X11 estar pronto — mas o `--headless` não precisa de X11, tudo bem. Problema real: se o usuário habilitar AMBAS as units, as duas sobem no login.

- **Correção:** install só ATIVA uma por vez. `hefesto daemon install-service [--headless]` troca a unit ativa. Ou units são mutuamente exclusivas via `Conflicts=` no `[Unit]`:
  ```ini
  Conflicts=hefesto.service          # na headless
  Conflicts=hefesto-headless.service # na normal
  ```

## 3. LACUNAS OPERACIONAIS NOVAS

### 3.1 FakeController: como gerar `hid_capture_{usb,bt}.bin`?

Decisão 3.6 e Patch 7 dependem de capturas HID pré-gravadas em `tests/fixtures/`. Não há procedimento documentado nem sprint responsável pela captura.

- **Proposta:** Sprint 1.1 inclui `scripts/record_hid_capture.py` que usa pydualsense em modo raw e grava 30s de input + state. Usuário com device roda uma vez em modo USB e uma em BT, commita binários (~500KB cada). Sem isso, ninguém sem device consegue escrever testes reprodutíveis.

### 3.2 Captures binários no repo — tamanho e Git LFS?

Se cada captura tem ~500KB, 2 arquivos, 1MB total = ok sem LFS. Se futuros cases (trigger effects específicos) adicionarem mais, pode passar de 10MB. Decidir cedo:

- **Recomendação:** não usar LFS. Manter abaixo de 5MB total. Se passar, reconsiderar.

### 3.3 `CHECKLIST_MANUAL.md` citado mas não listado no W0.1

Decisão 3.6 cria `CHECKLIST_MANUAL.md` pra smoke tests com device físico. O item 16 do Patch 1 não inclui esse arquivo.

- **Correção:** adicionar `CHECKLIST_MANUAL.md` no Sprint 0.1 com template: "Para cada DoD que requer device, usuário com hardware marca [x] antes de liberar release".

### 3.4 PyGObject build deps no bootstrap

Pyproject tem `[tray]` opcional com `PyGObject>=3.44`. Em Ubuntu 22.04 precisa `libgirepository1.0-dev`, `libcairo2-dev`, `pkg-config`, `python3-dev`. Bootstrap (item 9 do Patch 1) não cobre.

- **Dúvida:** instalar sempre (bootstrap mais pesado) ou só se usuário passar `--with-tray` pro `dev_bootstrap.sh`? Recomendo flag opt-in.

### 3.5 `setup_issues.sh` citado mas não no W0.1

Ordem de execução (linha 338 do HEFESTO_DECISIONS.md) item 5: "Abrir 26 issues via `scripts/setup_issues.sh` (criado em W0.1)". Mas o script não aparece na lista de 16 itens do Patch 1.

- **Correção:** adicionar `scripts/setup_issues.sh` no item 16 do Patch 1. Script tem array de 26 títulos/labels/bodies e chama `gh issue create` em loop.

### 3.6 `uaccess` exige systemd-logind

Patch 5 usa `TAG+="uaccess"`. Funciona em qualquer distro com systemd + logind (99% dos alvos: Pop!_OS, Ubuntu, Fedora). **Não funciona** em Gentoo OpenRC, Void runit, Alpine (OpenRC).

- **Dúvida:** escopo v0.x é "Linux com systemd"? Documentar limitação no README ou manter linha de código legado com `MODE="0666"` como fallback pra distros sem logind?
- **Recomendação:** documentar limitação. 99% dos usuários de DualSense em Linux estão em distros mainstream com systemd. Não complicar.

### 3.7 Debounce battery report (ADR-008)

Patch 7 linha 296: "Battery report com debounce de 5s pra evitar spam via USB". Não diz se é debounce de log ou de evento publicado no event bus.

- **Dúvida:** qual? Recomendo debounce no evento (`battery_change` só dispara se delta ≥ 1% OU passou 5s). Log segue regra normal.
